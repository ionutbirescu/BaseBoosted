We thought of creating a database suitable for storing data about songs, artists and their albums which can be used by a music providing application. We created the following tables: “Albums”, “Artists”, “Events”, “EventsLinks”, “Playlists”, “Songs” and “Users”. The “Albums”

table contains possible information about stored albums. Each Album has linked an artist by an “ArtistId” which is the foreign key between “Albums” and “Artists” table. The “Artists” contains information about possible artists such as their id, their name, their popularity rates, their number of listeners, their status, country origin or their type (whether Solo, Band, DJ). The “EventsLinks” contains a many-to-many relation between the “Events” table and “Artists” table, making the “Events” table more conclusive and without duplicate data. The “Songs” table

contains data regarding the Songs’ names and their artists and their albums which they appear on. “Users” table contains user-related data, likely including usernames, emails, subscription type or favorite artist.

The music database is useful because it can store multiple fields about songs, artists and users and link them to make it a good application for playing music. It stores the necessary data about the users and the music they want to listen to, making the music database well-suited for a music streaming service, enabling features like: Song and album search by artist, type, or popularity, the ability to explore new music and upcoming events based on users' preferences or social and sharing features where users can follow artists or share playlists.


Rares: I created and populated the Artists, Albums, Songs and EventsLinks table. I created 4 pages, 2 classic reports, 1 classic report with data aggregation functions and 1 chart. The pages have different functionalities such as displays all the DJs and Solo artists or displaying informations about the artists' albums number based on criterias.





Ionut: In this project, I worked on creating the userssongs, playlists, and events tables, along with their data scripts, which were rigorously tested for accuracy. I developed four web pages to enhance the application's functionality: List Certain Artists Page (Artists50Mil), which displays artists with more than a specified number of listeners in descending order; Searching Artists' Songs Page (ArtistSearch), an interactive report displaying all songs of a searched artist; Venue Counter Page (VenueCount), a summary table counting and sorting events by venue; and Number of Artists from Different Countries Page (ArtistCountry), a chart visualizing artist counts by country. Each page was designed to provide meaningful insights and interactivity to users.

Additionally, the database scripts were tested thoroughly to ensure schema integrity and data reliability. This work brought together database design and web application development to create a cohesive, functional system.

This is the Database schema:

drop table Artists cascade constraints; drop table Albums cascade constraints; drop table Songs cascade constraints; drop table Events cascade constraints;

drop table USERSSONGS cascade constraints; drop table EventsLinks cascade constraints; drop table Playlists cascade constraints

CREATE TABLE Artists (

AID VARCHAR2(10 BYTE) NOT NULL, ARTISTNAME VARCHAR2(40 BYTE) NOT NULL, LISTENERS NUMBER NOT NULL,

ABNO NUMBER,

STATUS CHAR(20 BYTE) NOT NULL, AWRDCNT NUMBER,

COUNTRY CHAR(40 BYTE) NOT NULL, TYPE CHAR(40 BYTE) NOT NULL,

PR NUMBER,

CONSTRAINT PK_Artists PRIMARY KEY (AID)

);


CREATE TABLE Events (

EVENTID VARCHAR2(10 BYTE) NOT NULL, NAME VARCHAR2(100 BYTE) NOT NULL, EVENTDATE DATE NOT NULL,

VENUE VARCHAR2(100 BYTE) NOT NULL, CITY VARCHAR2(100 BYTE) NOT NULL, COUNTRY VARCHAR2(50 BYTE) NOT NULL,

CONSTRAINT PK_Events PRIMARY KEY (EVENTID)

);


CREATE TABLE EventsLinks (

EVENTID VARCHAR2(10 BYTE) NOT NULL, ARTISTID VARCHAR2(10 BYTE) NOT NULL,

CONSTRAINT PK_EventsLinks PRIMARY KEY (EVENTID, ARTISTID), CONSTRAINT FK_Artist FOREIGN KEY (ARTISTID) REFERENCES Artists (AID),

CONSTRAINT FK_Event FOREIGN KEY (EVENTID) REFERENCES Events (EVENTID)

);


CREATE TABLE Albums (

ALBID VARCHAR2(10 BYTE) NOT NULL, TITLE VARCHAR2(40 BYTE),

AID VARCHAR2(10 BYTE), RLSYEAR NUMBER,

LABEL VARCHAR2(40 BYTE), TRKCNT NUMBER DEFAULT 0,

CONSTRAINT PK_Albums PRIMARY KEY (ALBID)

);


CREATE TABLE Playlists (

PLAYID VARCHAR2(10 BYTE) NOT NULL, NAME VARCHAR2(100 BYTE) NOT NULL, NOSONGS NUMBER NOT NULL,

USERID VARCHAR2(10 BYTE) NOT NULL, CREATEDAT DATE NOT NULL,

CONSTRAINT PK_Playlists PRIMARY KEY (PLAYID)

);


CREATE TABLE Songs (

SONGID VARCHAR2(10 BYTE) NOT NULL, SONGNAME VARCHAR2(40 BYTE) NOT NULL, RLSDATE DATE,

AID VARCHAR2(20 BYTE) NOT NULL, ALBID VARCHAR2(20 BYTE) NOT NULL,

GNR CHAR(40 BYTE) NOT NULL, DRT NUMBER NOT NULL,

CONSTRAINT PK_Songs PRIMARY KEY (SONGID)

);


CREATE TABLE USERSSONGS (

USERID VARCHAR2(10 BYTE) NOT NULL, NAME VARCHAR2(50 BYTE) NOT NULL, EMAIL VARCHAR2(100 BYTE) NOT NULL, COUNTRY VARCHAR2(50 BYTE), SUBSCRIPTIONTYPE VARCHAR2(20 BYTE), JOINDATE DATE NOT NULL,

FAVARTIST VARCHAR2(10 BYTE) NOT NULL, CONSTRAINT PK_Users PRIMARY KEY (USERID)

);
